# ChenTingToDo 全站代码与产品审查报告

审查日期：2026-07-18  
审查范围：本地 `main`（`e56cf2d`）、Vercel 生产站 `https://chentingtodo.vercel.app`  
审查方式：逐文件阅读、生产构建、依赖审计、线上桌面/移动端检查、关键交互验证、响应头检查

## 一、先说结论

这个项目的基础方向是正确的：它是一个很小的 Next.js 静态站，内容来自仓库内 Markdown，没有数据库、登录、写接口和服务端业务，部署在 Vercel。这样的架构便宜、稳定、攻击面小，也适合个人长期维护。

但它目前更像“视觉完成的 MVP”，还不是“可以低成本持续经营的网站”。最核心的问题不是缺少更多展示功能，而是：

1. **发布文章仍然很费事**：需要手写 frontmatter、创建文件、提交 Git、等待部署，没有模板、校验、草稿预览或一键发布脚本。你忙的时候自然不会更新。
2. **已有功能存在真实断点**：首页大搜索框点击无效；顶部搜索打开后线上没有搜索索引；标签和分类链接带了 URL 参数，但文章页不读取参数；文章卡片嵌套链接触发 React hydration 错误。
3. **安全基线欠维护**：Next.js 14.2.35 已被 `npm audit` 报告多个高/中风险公告；Markdown 明确关闭清洗后直接注入 HTML；生产响应没有 CSP 等基础安全头。
4. **内容和产品层级失衡**：只有 4 篇文章、2 个项目，却展示了搜索、分类、标签、分页、项目专区等一整套“内容很多时才有价值”的功能。网站当前最缺的是新内容和低摩擦发布，不是再加复杂功能。

综合评价：

| 维度 | 评价 | 小结 |
|---|---:|---|
| 架构选择 | 8/10 | 纯静态方案很适合个人站 |
| 代码可读性 | 7/10 | 结构清楚、组件不复杂，但缺少数据校验和类型收口 |
| 功能完成度 | 4/10 | 多个核心入口看起来存在、实际没有闭环 |
| 安全基线 | 5/10 | 攻击面小是优势，但依赖、HTML 清洗和响应头需要补齐 |
| UI 基础 | 7/10 | 清爽、响应式基本合格，但层级单一、信息密度与内容量不匹配 |
| 可持续运营 | 3/10 | 发布成本过高，是当前最大的产品问题 |

## 二、第一性原理分析

个人技术站的根本目标不是“功能齐全”，而是完成一个长期循环：

> 低成本记录 → 可靠发布 → 容易回看/检索 → 形成积累 → 愿意继续记录

当前网站把较多精力花在了“访问者如何浏览很多内容”上，但真正的瓶颈发生在第一步——站长如何快速产生并发布内容。只有 4 篇文章时，搜索、双层筛选和加载更多带来的价值很低；如果每次发布仍要手工处理多个步骤，再漂亮的前台也无法形成内容复利。

所以优化顺序应该是：

1. 先修复损坏的核心路径和安全问题。
2. 再把“写一篇并上线”的操作压缩到 1～2 个动作。
3. 然后减少低价值 UI，突出最近更新和真实项目进展。
4. 内容增长后，再逐步恢复或增强搜索、筛选、RSS 等能力。

## 三、已确认的问题（按优先级）

### P0：上线前应优先处理

#### F-01 首页主搜索框完全无效

- 位置：`src/app/SearchButton.tsx:3-12`、`src/components/Header.tsx:16-36`
- 证据：`SearchButton` 派发 `open-search` 自定义事件并设置一个从未用于渲染的 `open` 状态；`Header` 没有监听这个事件。
- 线上验证：移动端和桌面端点击首页“搜索文章...”按钮后，搜索框没有出现。
- 影响：首页 Hero 区最显眼的 CTA 是坏的，直接损害用户对网站完成度的判断。
- 最小修复：只保留一个搜索状态来源。可把 SearchDialog 提升到共享 Context，或让首页按钮直接调用 Header 暴露的打开方法。若暂时不修，先删掉首页重复搜索框，只保留顶部入口。

#### F-02 线上顶部搜索没有索引

- 位置：`src/components/SearchDialog.tsx:18-35`、`package.json:5-10`
- 证据：组件依赖 `/pagefind/pagefind.js`，但构建脚本只有 `next build`，没有执行 `pagefind --source out`。
- 线上验证：输入 `LangGraph` 后显示“搜索索引尚未生成”。
- 额外证据：本轮 `next build` 后 `out/pagefind/pagefind.js` 不存在；旧的本地产物曾存在索引，说明它来自手工命令而非可靠构建链路。
- 最小修复：添加 `postbuild`（例如 `pagefind --site out`，以当前版本命令为准）并在 Vercel 构建中验证产物。不要让生产 UI 向普通访问者展示开发者命令。

#### F-03 文章卡片存在嵌套链接，触发 hydration 错误

- 位置：`src/components/ArticleCard.tsx:14-57`、`src/components/TagChip.tsx:12-17`
- 证据：整个卡片由 `<Link>` 包裹，卡片内部每个标签又渲染为 `<Link>`，形成 `<a>` 嵌套 `<a>` 的非法 HTML。
- 线上验证：首页控制台连续出现 React 418/423 hydration 错误。
- 影响：服务端 HTML 与浏览器修复后的 DOM 不一致，可能导致点击行为、键盘可访问性和后续 React 更新不稳定。
- 最小修复：不要让整卡和标签同时都是链接。推荐让标题成为文章链接，卡片本体使用普通 `<article>`；标签保留独立链接。

#### F-04 标签和分类 URL 不生效

- 位置：`src/components/TagChip.tsx:14`、`src/app/articles/[slug]/page.tsx:65-74`、`src/app/articles/ArticlesList.tsx:14-28`
- 证据：链接写入 `?tag=` 或 `?category=`，但 `ArticlesList` 只使用 `useState(null)`，没有读取 `searchParams`。
- 线上验证：访问 `/articles?tag=Docker` 仍显示 4 篇文章，且没有选中任何标签。
- 最小修复：用 `useSearchParams` 初始化并同步筛选状态，或彻底取消可点击标签/分类，只保留纯文本标签。

#### F-05 GitHub 链接指向不存在的仓库

- 位置：`src/components/Footer.tsx:19-26`、`src/app/articles/[slug]/page.tsx:133-144`、`content/projects/chen-ting-todo.md:5`
- 证据：代码指向 `github.com/chenting/todo`；实际仓库是 `github.com/ChenTingToDo/chentingtodo`，前者无法通过 GitHub API解析。
- 影响：页脚“GitHub”、项目源码和“编辑此页”均可能进入 404。
- 最小修复：统一定义站点仓库常量，避免三处硬编码漂移。

### P1：本轮维护应处理

#### S-01 Markdown 可形成存储型 XSS（条件性中风险）

- Rule ID：NEXT-XSS-001 / REACT-XSS-001
- 严重性：Medium（当前仓库仅由你维护时）；若未来接入投稿、AI 自动写稿或 CMS，则提升为 High
- 位置：`src/components/MarkdownContent.tsx:21-25`
- 证据：`remarkHtml` 配置为 `{ sanitize: false }`，结果随后进入 `dangerouslySetInnerHTML`。实测恶意 Markdown 会原样输出 `<img onerror=...>` 和 `javascript:` 链接。
- 影响：一旦恶意或被污染的 Markdown 进入仓库并部署，访问文章即可执行任意同源脚本。
- 修复：默认禁止原始 HTML，并加入 `rehype-sanitize` 严格白名单；同时限制链接协议为 `http/https/mailto`。不要依赖“这是自己的文章”作为长期安全边界。
- 缓解：在修复渲染器之前，不要把网页抓取结果、第三方投稿或未经审阅的 AI 内容直接发布。
- 误报边界：当前没有在线投稿接口，因此外部攻击者不能直接提交内容；这是供应链/内容管线风险，不代表网站现在已被入侵。

#### S-02 Next.js 依赖线已过旧并命中安全公告

- Rule ID：NEXT-SUPPLY-001
- 严重性：High（维护优先级）；当前静态部署使部分服务端 DoS/SSRF 公告不可利用
- 位置：`package.json:12-21`、`package-lock.json`
- 证据：实际构建版本为 Next.js `14.2.35`；`npm audit` 返回 1 个 high、2 个 moderate 依赖问题，Next.js 条目包含 RSC DoS、缓存投毒、请求走私、SSRF 和部分 XSS 公告。
- 影响：继续停留在旧主版本会累积已知漏洞和升级成本。
- 修复：单独开升级任务，升级到当前受支持且已修补的 Next.js 版本；不要直接执行 `npm audit fix --force` 后上线。升级后重跑构建、页面路由、搜索和 hydration 检查。
- 误报边界：当前 `output: 'export'` 且没有 API/中间件/图片优化服务，许多服务端漏洞路径不在生产暴露面中；风险重点是版本维护和未来架构变化。

#### S-03 Markdown/搜索 HTML 注入缺少 CSP 兜底

- Rule ID：NEXT-CSP-001 / NEXT-HEADERS-001
- 严重性：Medium（与 S-01 组合后风险更高）
- 位置：`next.config.js:1-10`、`src/app/layout.tsx:29-44`
- 证据：生产站主响应有 Vercel HSTS，但未观察到 CSP、`X-Content-Type-Options`、`X-Frame-Options`/`frame-ancestors`、`Referrer-Policy` 和 `Permissions-Policy`。页面另有内联主题脚本。
- 影响：发生 HTML 注入时没有浏览器侧脚本执行限制；页面也可被第三方 iframe 嵌套。
- 修复：静态导出可通过 `vercel.json` 设置安全头；先把主题初始化脚本外置或计算固定 hash，再启用不含 `unsafe-eval` 的 CSP。
- 误报边界：HSTS 已由 Vercel 提供，不需要重复配置；响应头可能由后续平台设置改变，应以上线实测为准。

#### S-04 搜索摘要再次直接注入 HTML

- Rule ID：REACT-XSS-001
- 严重性：Low（Pagefind 本地索引、内容可信时）
- 位置：`src/components/SearchDialog.tsx:139-157`
- 证据：`result.excerpt` 使用 `dangerouslySetInnerHTML`。
- 影响：若索引内容或 Pagefind 返回值可被污染，会形成第二个 HTML 注入点。
- 修复：确认 Pagefind 的 excerpt 安全契约并锁定版本；更稳妥做法是只允许 `<mark>`，对其余标签和属性进行白名单清洗。

#### Q-01 缺少内容运行时校验

- 位置：`src/lib/content.ts:47-68`、`107-128`
- 证据：frontmatter 直接 `as ArticleFrontmatter/ProjectFrontmatter`，TypeScript 断言不校验真实 Markdown 数据。
- 影响：日期、tags、tech 或 published 写错后，可能在构建时崩溃、排序异常或意外发布。当前逻辑是 `published !== false`，字段缺失也会发布。
- 修复：使用轻量 schema（如 Zod）在构建时校验，错误信息包含文件名；发布条件改为 `published === true`。

#### Q-02 工程校验链不完整

- 位置：`package.json:5-10`、仓库根目录
- 证据：没有测试文件、没有 CI；`npm run lint` 首次运行进入交互式 ESLint 配置，不能用于自动检查；README 只有一个标题。
- 构建结果：`npm run build` 最终成功，但多次提示 lockfile 缺少 SWC 依赖并无法自动修补。
- 影响：错误只能在用户访问或部署后被发现，维护间隔越长越容易忘记正确流程。
- 修复：补 ESLint 配置、内容校验、最小烟测和 GitHub Actions；重新生成一致的 lockfile。

#### A-01 搜索弹窗可访问性不足

- 位置：`src/components/SearchDialog.tsx:81-171`
- 证据：弹窗没有 `role="dialog"`、`aria-modal`、可访问标题、明确关闭按钮或焦点陷阱；关闭后也没有把焦点还给触发按钮。
- 影响：键盘和屏幕阅读器用户容易迷失，背景内容仍处在可访问树中。
- 修复：优先使用原生 `<dialog>` 或成熟的无头组件；补焦点管理和可见关闭按钮。

### P2：内容增长后再做

- 阅读时长按空格切词，中文正文会被严重低估：`src/app/articles/[slug]/page.tsx:40-42`。
- 全站缺少 `metadataBase`、canonical、OG 图片、sitemap、robots 和 RSS；这影响分享与长期订阅，但优先级低于持续更新。
- 表格只有 `width: 100%`，没有移动端横向容器，未来宽表可能溢出。
- 搜索结果和筛选状态使用 `any`，增加维护风险，但不是当前用户故障的首因。

## 四、哪些功能没什么用，可以删或收缩

这里的判断基于当前只有 4 篇文章、2 个项目，而不是认为这些功能永远无用。

### 建议现在删除/合并

1. **删除首页重复搜索入口，直到搜索完全修好**  
   顶部已有搜索按钮和快捷键。首页再放一个坏掉的大搜索框只会放大失败体验。修好后也建议二选一，不要两个入口表达同一动作。

2. **分类与标签只保留一层**  
   当前 4 篇文章出现 4 个分类和 14 个标签，筛选 UI 比内容还重。建议先保留标签，分类暂时变为卡片元信息；内容超过 20～30 篇后再恢复双层筛选。

3. **隐藏“加载更多”机制**  
   `visibleCount=10` 在当前永远不会显示按钮。代码可以保留，但没有必要继续围绕它设计。

4. **收缩 About 页硬编码内容**  
   “学习目标”和“技术栈”很容易过时，也与首页/项目页重复。建议 About 只保留：你是谁、网站为何存在、如何联系/关注、最近在做什么。

5. **项目区只展示真实且可访问的项目**  
   如果“AI Agent 实验框架”没有仓库、演示或真实进展，建议标为“实验/暂停”或暂时隐藏。空泛项目会稀释可信度。

6. **删除确定的死代码/无效状态**
   - `SearchButton` 的 `open` 状态和无人监听的 `open-search` 事件。
   - `src/lib/content.ts` 中未使用的 `getArticlesByCategory`、`getArticlesByTag`（或真正复用它们）。
   - 未使用 Pagefind UI 组件时，`globals.css` 中的 `--pagefind-ui-*` 变量。

### 建议保留

- 深色模式：实现成本已沉没，线上表现正常。
- 静态 Markdown 架构：不要为了“后台上传”立刻引入数据库、账号系统和 CMS。
- 项目页：它能体现真实产出，但内容必须保持真实、可验证、及时更新。
- 搜索：长期有价值，但前提是纳入自动构建；修不好就先隐藏。

## 五、最值得添加什么

### 1. 一条低摩擦发布流水线（最高价值）

不要先做在线后台。最小可行方案是：

- `npm run new:article -- "标题"`：自动生成日期、slug、frontmatter 和草稿文件。
- `npm run check:content`：校验字段、日期、重复 slug、危险 HTML、失效链接。
- `npm run preview`：本地预览草稿。
- `npm run publish -- slug`：把 `published` 改为 true，执行构建与搜索索引检查，给出可提交结果。
- 在仓库提供 `content/templates/article.md` 和 10 行以内的中文发布说明。

这样仍保留静态站的安全和低成本，同时把发布从“记住多个工程步骤”变成“运行一条命令”。

如果你主要用手机记录，第二阶段可以增加 GitHub Issue 模板或 Notion → PR 自动化，但自动生成的 PR 必须经过内容清洗和人工确认后才能发布。

### 2. 内容状态，而不是更多栏目

增加 `status: seedling | growing | evergreen` 或简单的“草稿/持续更新/定稿”状态，比当前按 30 天自动显示 `New` 更符合数字花园。当前所有文章超过 30 天后，`New` 逻辑已完全失去展示价值。

### 3. 最近更新与站点活性

- 首页优先展示“最近更新”，不只看最初发布日期。
- 项目卡片显示状态：进行中 / 暂停 / 已完成。
- 如果长期未更新，不伪装活跃；明确写“最近更新时间”反而更可信。

### 4. RSS、sitemap 和基础分享卡片

它们比评论、点赞、账号系统更适合个人知识站：维护成本低，能带来长期分发与可发现性。

### 不建议近期添加

- 登录/注册、在线评论、点赞、复杂后台、数据库全文搜索、AI 聊天机器人、访问者画像。
- 原因：这些功能不解决“你没时间上传文章”的根因，却显著增加安全、隐私和维护成本。

## 六、UI 审查

### 做得好的地方

- 桌面端和 390px 移动端都没有横向溢出，基础响应式合格。
- 留白、绿色主题、卡片和深色模式形成了统一视觉语言。
- 导航、文章列表、项目列表的结构清晰，首次访问能理解站点是什么。
- 移动端头部压缩合理，主要内容仍可读。

### 需要优化的地方

1. **首屏信息重复且过高**  
   Logo、站名、描述、口号、搜索框连续出现，桌面端首屏大量空间用于重复品牌表达。建议把 Hero 高度减少约 30%，让最近更新更早进入视野。

2. **所有内容都像同一种卡片**  
   文章、项目、筛选按钮、口号都依赖浅绿边框/圆角，视觉层级不足。建议卡片默认用中性边框，绿色只用于交互状态、重点和链接。

3. **中文标题使用等宽字体过多**  
   等宽字体适合英文品牌和代码，不适合大段中文标题；当前字形和间距显得机械。建议品牌名/代码保留 mono，中文标题使用系统无衬线或一款稳定中文字体。

4. **文章页缺少长文导航能力**  
   未来文章变长后，应优先增加桌面端目录、标题锚点和移动端阅读进度，而不是更多首页装饰。

5. **移动端品牌识别偏弱**  
   390px 下左上角只显示一个 `C`。可保留紧凑布局，但建议用更有辨识度的图形/字标，或者显示缩短后的 `ChenTing`。

6. **筛选区对当前内容量过重**  
   14 个标签按钮在移动端会占据大块空间。建议折叠到“筛选”抽屉，或只显示使用频率最高的 5 个。

7. **交互反馈与语义不一致**  
   项目卡片右上角是“外链”图标，但点击卡片实际进入站内项目详情；应换成普通箭头或删除图标。

## 七、建议执行顺序

### 第一批：修复可信度和安全底线（约 1 个小版本）

1. 修首页搜索、Pagefind 构建、标签 URL、嵌套链接和错误 GitHub 地址。
2. 清洗 Markdown，限制危险 URL；补基础安全响应头。
3. 升级 Next.js 和依赖，修 lockfile。
4. 补非交互式 lint、内容 schema 校验和最小 CI。

验收标准：

- 首页两个搜索入口若都保留，则都能打开同一个弹窗并返回结果。
- `/articles?tag=Docker` 只显示 Docker 相关文章并显示选中状态。
- 生产控制台没有 hydration 错误。
- `npm run build` 后必然生成 Pagefind 索引，且无 SWC lockfile 警告。
- 恶意 Markdown 的事件属性和 `javascript:` 链接不能进入最终 HTML。
- `npm audit` 不再保留可修复的 high 风险项。

### 第二批：降低发布成本

1. 增加文章模板、创建命令、内容校验和发布命令。
2. README 写成一页中文操作手册。
3. 增加草稿预览和站点构建 CI。

验收标准：从一个标题开始，5 分钟内能生成合规草稿；发布时只需一个明确命令和一次 Git 提交。

### 第三批：UI 收缩与内容重心调整

1. 缩短 Hero、减少绿色边框、降低筛选区存在感。
2. About 页去除易过时硬编码；项目增加真实状态。
3. 内容增长后再加入 RSS、sitemap、目录和分享卡片。

## 最终小结

网站不需要推倒重做。最合适的策略是“小修核心、砍掉伪需求、先让内容流动起来”。

如果只允许做一件事，应先完成“文章创建 → 校验 → 预览 → 发布 → 自动生成搜索索引”的单命令流水线；如果允许做第二件事，再修复当前 5 个 P0 功能问题。等内容真正积累后，搜索、筛选、目录和 RSS 才会自然产生价值。
