# ChenTingToDo 文章结构与发布流程简化计划书

日期：2026-07-18
阶段：方案设计（本阶段不修改文章正文与网站发布逻辑）

## 一、目标与成功标准

这次工作的目标不是再造一个“博客后台”，而是让一篇已经写好的 Markdown 从归档文件变成网站文章时，不再依赖记忆多项格式和 Git 操作。

成功标准：

1. 任何一篇现有 Markdown 都能通过一个明确入口导入，不再手工复制、改名和补齐全部元数据。
2. 系统能在发布前指出缺少的字段、错误的标题层级、危险 HTML、重复 slug 和明显失效的内部链接。
3. 草稿可以在真实网站样式中预览，但不会被正式构建意外发布。
4. 从“文章正文已完成”到“得到可推送版本”只需要一个发布命令。
5. 发布前自动完成内容校验、站点构建和 Pagefind 搜索索引生成；失败时不推送。
6. 操作说明控制在一页中文文档内，隔几个月再回来也不需要重新学习。

## 二、已确认的现状

### 1. 文章现状

本次检查了 `G:\ChenTingToDo-文章归档` 中的 7 篇文章。它们的共同优点是：

- 都有清楚的主题和时间线；
- 大多能交代事情为什么发生、做了什么、最后得到什么；
- 语言较易读，已经具备个人复盘文章的雏形；
- 每篇都有“下一步”或“经验总结”，适合数字花园的持续积累定位。

但它们目前更接近“GPT 根据对话整理出的总结稿”，还不是可直接发布稿：

- 7 篇都没有网站当前必需的 frontmatter；
- 正文先使用 `##`，随后大量回到 `#`，标题层级不规范；
- 很多“问题”和“结果”只有概括，没有保留精确报错、命令、配置、证据或复验结果；
- “经验总结”常使用“第一、第二、第三”，观点正确但比较模板化，个人判断和具体证据偏少；
- 网站从零上线、Vercel、域名、Search Console 四篇之间存在一定内容重复；
- 部分技术和工具信息具有时效性，正式发布前需要核对当前事实；
- 文章类型不同，不能强制全部套用“报错复盘”结构。

### 2. 分篇判断

| 文章 | 更合适的类型 | 当前优点 | 发布前主要补充 |
| --- | --- | --- | --- |
| Google Search Console 配置 | 配置教程 / 故障复盘 | 背景、验证失败、最终结果齐全 | 精确验证提示、等待时长、最终验证证据 |
| Codex + Skills + MCP | 学习记录 / 环境搭建 | 概念和一次网络故障都有覆盖 | 实际命令、原始错误、当时与当前工具状态核对 |
| Obsidian 知识库 | 学习记录 | 认知变化和概念路径清楚 | 真实库结构、实际操作、产出证据；不必虚构“报错” |
| 网站从零上线 | 项目复盘 | 项目动机、选型和结果完整 | 避免与三篇专项文章重复；突出总览和关键决策 |
| Vercel 部署踩坑 | 故障复盘 | 七篇中最接近完整教程 | 精确构建日志、错误配置与正确配置、复验步骤 |
| 自定义域名配置 | 配置教程 / 决策记录 | 选择过程和 DNS 学习路径清楚 | 实际记录类型、传播等待、HTTPS 与主域验证 |
| SmartRecipes 产品设计 | 产品决策记录 | 用户问题、取舍、MVP 边界较完整 | 把“设计完成”与“产品已验证”分开，补验收假设和待验证项 |

### 3. 当前网站发布链路

网站现在是 Next.js 静态导出：

```text
Markdown 文件
  → 手写 frontmatter
  → 放入 content/articles
  → Git 提交并推送
  → Vercel 构建
  → Pagefind 生成搜索索引
  → 正式站点
```

当前 frontmatter 依赖 `title`、`date`、`tags`、`category`、`description` 和 `published`，但代码只做 TypeScript 类型断言，没有对真实 Markdown 数据进行严格运行时校验。`published` 缺失时也会被当作可发布内容。

因此，真正的负担不是“上传速度”，而是以下五种成本叠加：

1. **记忆成本**：记住目录、字段、slug、命令和发布顺序。
2. **转换成本**：把聊天总结稿手工改成网站格式。
3. **检查成本**：人工发现格式、链接、元数据和敏感信息问题。
4. **切换成本**：在编辑器、终端、GitHub、Vercel 之间来回操作。
5. **失败成本**：错误通常到构建或线上才暴露，需要重新修复和部署。

## 三、第一性原理判断

个人知识网站最核心的循环是：

```text
真实经历
  → 低成本记录
  → 可复用文章
  → 可靠发布
  → 日后能搜索和回看
  → 更愿意继续记录
```

现在的瓶颈在“记录稿到可发布稿”之间，而不是前台缺少更多功能。因此方案应遵守四条原则：

1. **保留已经可靠的部分**：继续使用 Markdown、GitHub、Vercel 和静态站，不增加数据库、账号系统或 CMS。
2. **自动化确定性工作**：文件命名、frontmatter、格式校验、构建和索引检查交给脚本。
3. **保留需要判断的工作**：事实准确性、隐私、文章观点和最终是否发布仍由人确认。
4. **失败要早且可理解**：问题应在本地以中文、文件名和具体修复建议显示，而不是部署后才发现。

## 四、目标文章结构

### 1. 所有文章共享的元数据

```yaml
---
title: "文章标题"
date: "2026-06-03"
updated: "2026-07-18"
tags: ["Vercel", "Next.js"]
category: "问题复盘"
description: "一句话说明遇到了什么、解决了什么。"
published: false
---
```

其中 `updated` 可以作为第二阶段字段；第一阶段如要保持最小改动，可以暂不增加。

### 2. 不采用一个万能正文模板

#### A. 故障复盘 / 技术教程

适用于 Vercel、DNS、Search Console、网络或开发报错：

```text
背景与目标
环境与前置条件
现象与原始报错
排查过程（做了什么 → 结果 → 得到什么判断）
根因
最终方案
验证方法与结果
容易踩坑的点
参考资料
```

#### B. 学习记录

适用于 Codex、Skills、MCP、Obsidian：

```text
为什么学习
希望解决的问题
核心概念
实际操作或练习
遇到的困惑与修正
形成的判断
可验证的产出
下一步
```

#### C. 项目 / 产品决策记录

适用于 ChenTingToDo 总览、SmartRecipes：

```text
背景与真实问题
目标、用户和约束
考虑过的方案
关键取舍及理由
最终设计或实施过程
当前结果
如何验证 / 尚未验证什么
后续计划
```

“验证”必须符合文章类型：技术文章验证修复是否生效，学习文章验证是否形成实际产出，产品设计文章验证假设和 MVP 标准。没有发生过报错时，不为模板完整而编造报错。

## 五、目标发布流程

### 1. 用户看到的流程

```text
把 Markdown 放入约定的 inbox 目录
  → 运行“准备文章”命令
  → 按提示只补充需要人工判断的内容
  → 在网站样式中预览草稿
  → 运行“发布文章”命令
  → 确认后提交并推送
  → Vercel 自动上线
```

建议最终提供以下命令：

```bash
npm run article:prepare -- "G:\路径\原稿.md"
npm run article:check -- slug
npm run article:preview -- slug
npm run article:publish -- slug
```

其中：

- `article:prepare`：读取原稿，生成安全 slug，补 frontmatter 草稿，规范基础标题层级，并写入草稿区；绝不覆盖已有文章。
- `article:check`：校验元数据、日期、slug、标题层级、危险 HTML、空链接、重复标题和常见敏感信息。
- `article:preview`：让 `published: false` 的指定草稿也能在真实文章页样式中预览。
- `article:publish`：再次校验，把文章切换为发布状态，执行完整构建和搜索索引检查，成功后输出待提交文件与下一条 Git 命令。

第一版不建议脚本自动 `git push`。发布脚本先生成“已验收、待提交”的结果，再由用户执行一次明确确认；稳定使用一段时间后，才考虑增加 `--push` 参数。

### 2. 草稿安全边界

- 新文章默认必须是 `published: false`。
- 网站读取逻辑改为只有 `published === true` 才进入生产构建。
- 草稿预览只在明确的本地预览模式启用。
- 自动整理只能做机械转换，不能擅自补写不存在的报错、验证结果或产品数据。
- 发布前显示隐私检查结果，重点关注密钥、Token、邮箱、内网地址、真实账号和不应公开的域名配置。

## 六、分阶段实施计划

### 阶段 0：先整理这 7 篇，不批量盲目发布

1. 为每篇文章确定类型、slug、分类和标签。
2. 补齐 frontmatter，并把正文标题统一为从 `##` 开始的递进结构。
3. 去掉模板化重复句，保留个人真实判断。
4. 把网站总览与 Vercel、域名、Search Console 专项文章做成“总览 + 关联阅读”，避免重复粘贴。
5. 对技术事实、链接、命令和隐私逐篇复核。
6. 先选择 Vercel 文章作为试点，跑通完整发布流程，再处理其余 6 篇。

验收：试点文章结构完整、事实可核对、本地页面正常、搜索可找到、生产构建通过。

### 阶段 1：最小可用发布工具

1. 新增三类正文模板和统一 frontmatter 模板。
2. 实现 `article:prepare` 与 `article:check`。
3. 修正生产发布条件为 `published === true`。
4. 为缺字段、错误日期、重复 slug 和危险内容增加可读错误信息。
5. 编写一页中文发布说明。

验收：从任意一篇原稿到合规草稿不超过 5 分钟；错误信息能直接定位到文件和字段。

### 阶段 2：真实预览与单命令发布验收

1. 实现只在本地启用的草稿预览。
2. 实现 `article:publish`：内容检查 → 切换发布状态 → lint/typecheck/test/build → Pagefind 索引确认。
3. 失败时恢复文章的草稿状态，不留下“半发布”结果。
4. 输出简洁的成功摘要和待提交文件，不自动覆盖或推送。

验收：发布失败不会进入 Git 提交；发布成功后文章页、列表、分类、标签和搜索均能找到该文章。

### 阶段 3：根据真实使用再决定是否增加入口

只有在本地命令仍然妨碍记录时，才选择一种额外入口：

- 经常在电脑写：增加右键 / 拖拽式导入或一个中文批处理入口。
- 经常在手机写：增加 GitHub Issue 模板，由自动化生成草稿 PR，人工确认后合并。
- 已经稳定使用 Notion：再考虑 Notion → 草稿 PR；不让 Notion 直接发布生产。

不同时建设多个入口，以免产生内容来源、格式和同步冲突。

## 七、建议的首轮实施范围

首轮只做以下内容：

1. 三类模板；
2. 原稿导入；
3. frontmatter 与正文结构校验；
4. 默认草稿和严格发布条件；
5. Vercel 文章试点；
6. 一页中文说明。

首轮明确不做：

- 在线 CMS；
- 数据库和用户登录；
- AI 自动生成事实、报错或验证结论；
- 手机端自动发布；
- 自动推送 Git；
- 同时重写全部 7 篇；
- 与文章发布无关的 UI 改版。

## 八、实施后的最终验收清单

- [ ] 原稿路径包含中文和空格时仍可正常导入。
- [ ] 同名或同 slug 文章不会被覆盖。
- [ ] 缺失 frontmatter 时给出中文、可执行的修复提示。
- [ ] `published: false` 永远不会进入生产文章列表和搜索索引。
- [ ] 草稿能以正式文章样式在本地预览。
- [ ] 标题层级错误、危险 HTML 和明显敏感信息会阻止发布。
- [ ] 发布命令会运行现有完整工程检查并验证 Pagefind 索引。
- [ ] 构建失败时文章仍保持草稿状态。
- [ ] Vercel 试点文章在线可访问，列表、标签和搜索结果一致。
- [ ] 中文发布说明能够让几个月没碰项目的人在 10 分钟内完成一次发布。

## 九、推荐结论

最值得优先做的不是“上传后台”，而是把现有仓库变成一个不会让人记步骤的发布工具。先用 Vercel 踩坑文章验证“导入 → 校验 → 预览 → 发布”闭环；闭环可靠后，再逐篇整理其余文章。这样每一步都能产生可见收益，也不会为了降低七篇文章的发布成本，反而增加一套长期维护负担。
